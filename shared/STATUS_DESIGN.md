- **接口要求**:输出可被 Orient 切片消费的【动量观察】

### Block 3:基底层(IH44 & IF 纯净)

- **接口要求**:输出可与动量层对比的【基底观察】

### Block 4:波动率组块

- **理念**:波动率层面【该升不升 / 该降不降】的状态异常
- **接口要求**:输出可被 Orient 切片消费的【波动率观察】

### Block 6:量价 & HHI 组块

- **理念**:成交额的【缩圈】程度反映拥挤度
- **接口要求**:输出可被 Orient 切片消费的【拥挤度观察】

## 5. Orient 切片与准备态 ⭐

### 设计思路(三层委托)

```
Orient 切片(线上职责)
   │  我们要做什么判断
   ▼
准备态(线上 + 本地共同明确)
   │  数据需要准备到什么程度
   │  · 输入数据来源
   │  · 计算口径
   │  · 状态格式
   │  · 历史窗口
   ▼
具体切片阈值(本地完善)
   │  临界值是多少
   │  触发规则是什么
   │  本地推测/测试/迭代
```

### 为什么这样设计

- **Orient 切片**是【判断框架】的需求方,由线上副脑明确
- **准备态**是【数据→分析】的衔接,共同明确,标准化便于流式范式
- **阈值**是【经验/测试】的产物,本地最了解数据特性,由本地推测完善
- 这种分工避免:线上越界给阈值(过拟合不可靠),本地猜测切片(框架被打散)

### 准备态的共通约束(所有切片)

无论哪个 Orient 切片,准备态都遵循:

```
1. 流式范式
   · 状态读取 → 增量更新 → 偏离累积 → 阈值检测 → 状态写回
   · 可重复跑、跑一次更新一天、可回测复现

2. 状态格式标准化
   · 每个切片有独立的 state 文件
   · 同一切片的 state 跨时间一致

3. 输出可被下游消费
   · 每个切片产生【信号值 + 状态标记】
   · 高阶切片(ABCD/三段式)可消费低阶切片输出

4. 准备态的【实现】由本地决定
   · 具体用什么库、什么数据结构、什么文件格式
   · 线上只定义【接口要求】,不规定实施方式
```

### 阶段划分(Phase 0 → 3)

- **Phase 0**:1 个切片(ATR),验证流式范式本身
- **Phase 1**:把框架 v2 三信号 + 见顶预警 v0.1 切片化
- **Phase 2**:把 ABCD / 三段式 等综合切片落地
- **Phase 3**:元命题层(双向四象限 / 三反馈分离)落地

## 6. Phase 0 实施 · ATR 切片的准备态

### ATR 切片的准备态(详细版)

```
输入数据:
  · 某指数的 OHLC 日线(本地已有)
  · 可选:OHLC 分钟线(已有)

计算口径:
  · ATR = avg(true_range over N days)
  · true_range = max(high-low, |high-prev_close|, |low-prev_close|)

历史窗口:
  · ATR 短窗口:3 / 5 / 7 中选(待本地测试)
  · ATR 长窗口:14 / 20 / 30 中选(待本地测试)
  · 历史分布回看:60 / 100 / 200 天中选(待本地测试)

状态格式(state 文件):
  · 单文件,记录每个指数的当前 ATR 值
  · 含字段:index_code, date, atr_short, atr_long, 
    ratio (= short/long), distribution_percentile, 
    cusum_value, last_trigger_date
```

### 比对方式(几种选择,待本地+99 决策)

```
1. 静态阈值比对(最简,但难适应市场)
   · ATR 当前值 vs 固定阈值

2. 动态阈值比对
   · ATR 当前值 vs 滚动 N 天均值

3. 历史分布对比(推荐起点)
   · ATR 当前值在过去 N 天分布中的【分位】
   · 比如 95 分位 → 触发

4. 多窗口对比(框架更通用)
   · ATR(短)/ ATR(长)的【比值】
   · 反映短期偏离长期的程度
```

### 数据存放态(几种选择)

```
1. JSON 文件(推荐起点)
   · 简单、可读、易调试
   · 适合状态量小

2. 内存状态 + 持久化文件
   · 运行时内存 + 关停时写
   · 适合脚本式工具

3. SQLite
   · 单文件、可查询、并发友好
   · 适合状态历史需要回溯查询

4. Parquet / DuckDB
   · 列式、高效
   · 适合 Phase 1+ 多 Block 聚合,起点不需要
```

### Phase 0 验收锚点

- ATR + CUSUM 能复现 5/12 那次见顶预警
- 不是【事后回看命中】,而是【算法本身能流式检测出】

### Phase 1+ 推广

ATR 切片跑通后,把 § 5 表格里 Phase 1 的切片逐个落地:
- 缩圈 HHI / 风格 spread / 承接失三件套
- 每一个都遵循同样的【切片 → 准备态 → 阈值】链路
- 共享同一套流式范式 + state 格式标准

## 7. 本 Project 范围与边界

### 做什么

- fact 层面的【数据采集-预处理】(已有 ~15,000 CSV)
- Orient 切片的【清单 + 准备态】定义(§ 5)

- truth Project 的信息源采集

### 接口怎么留

- **Observe 层 #1 接口**:entity 输入位
- **Orient 层 #2 接口**:叙事阈值评价位
